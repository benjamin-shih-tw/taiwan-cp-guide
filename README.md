# 🎯 Notion「coding course」演算法與程式實作建議指南

本指南整合自 Notion 課程「**coding course**」實戰講義中收錄的關鍵優化技巧、防禦性程式設計（Defensive Programming）觀念，以及演算法競賽的避坑指南。以下依據不同演算法與資料結構模組整理出核心建議內容：

---

## 📂 核心建議內容

### 1. 前綴和與差分 (Prefix Sums & Difference)
*   **強烈建議使用 1-based 索引**：將前綴和與差分陣列設計為 $1$-indexed（長度為 $N+1$），並將 `pref[0]` 初始化為 `0`。
    *   *原因*：在處理區間端點 $L=0$ 時，`pref[R] - pref[0]` 不會發生陣列索引越界（Out of Bounds）的執行期錯誤（Runtime Error），能大幅簡化邊界條件判定。

### 2. 模擬與矩陣幾何 (Simulation & Matrix Geometry)
*   **統一變數命名規範**：在實作二維矩陣或網格模擬題時，一律使用 `r` (Row) 和 `c` (Column) 來命名變數，避免混用 `x` 與 `y`。
    *   *原因*：混用 `x` 與 `y` 極易在進行轉置（Transpose）或邊界判定時引發錯位 Bug。
*   **複雜狀態轉移設計**：在面對複雜的狀態模擬時，**建議在草稿紙上先畫出狀態轉移的流程圖**再動筆寫程式，切忌一邊想一邊寫。

### 3. 二分搜尋法 (Binary Search)
*   **實數二分搜尋的精確度防禦**：進行浮點數二分搜尋時，為了保險起見，**強烈建議直接使用固定次數的迴圈（如 100 次）**，而非依賴精度差值（`r - l > eps`）作為終止條件。
    *   *原因*：這能保證完美的精確度與執行效率，徹底杜絕因浮點數精度限制導致的死鎖或無窮迴圈問題。

### 4. 雜湊與碰撞防禦 (Hashing & Collisions)
*   **生日悖論與雙雜湊防禦**：單雜湊的碰撞概率在資料量 $N \approx \sqrt{M}$ 時會大幅上升。
    *   *原因*：為了在競賽中防止被刻意設計的測資卡掉，**建議採用雙雜湊**（例如開兩個不同的模數如 $10^9+7$ 和 $10^9+9$）進行防禦。

### 5. 樹與動態節點優化 (Tree & Dynamic Node Optimization)
*   **記憶體池優化 (Memory Pool)**：在 C++ 中動態使用 `new` 節點會引發大量記憶體碎片與嚴重的常數時間延遲。
    *   *原因*：為了優化效能，建議在實戰中使用**靜態陣列分配器模擬指標**（即手寫靜態記憶體池）來分配節點空間。

### 6. 點集與映射 (Sets & Maps)
*   **I/O 效能優化**：在涉及大量資料結構（如 `set`、`map`）的操作與頻繁輸出入時，競賽中建議開啟 **Fast I/O** 以提高 I/O 效率。
    *   *範例*：
        ```cpp
        ios::sync_with_stdio(0);
        cin.tie(0);
        ```
*   **指定初始容量**：針對大數據量的自訂 Hash Map 等資料結構，初始化時建議指定初始容量以優化效能並避免頻繁的 Rehash。

### 7. 貪婪演算法極值處理 (Greedy Algorithms)
*   **負數邊界值初始化**：在貪婪選擇中，若區間或數值可能為負數，直接初始化為 `0` 會導致負數區間被錯誤過濾。
    *   *建議*：在 C++ 中建議使用 `numeric_limits<int>::min()` 或 `-2e9`，Java 中使用 `Integer.MIN_VALUE`，Python 中使用 `-float('inf')`。

### 8. 位元運算與符號防禦 (Bitwise Operations)
*   **符號位元防禦**：在實作位元遮罩時，`mask` 應被視為無符號的狀態字。
    *   *建議*：建議使用 `>>>`（無符號右移，如 Java/JS）或確保 `mask` 為非負整數以防符號位元干擾位元運算。
